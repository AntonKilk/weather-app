# Plan: Installable PWA — Manifest, Icons, Service Worker (STORY-006)

## Summary

Wire `vite-plugin-pwa` into the existing Vite 7 + TypeScript-strict setup so the
production build emits a valid web app manifest, a registered service worker
that precaches the app shell, and the icons referenced from both. The user can
"Add to Home Screen" on iPhone Safari, the app opens standalone with the right
name/icon, and the cached shell loads in airplane mode. No new domain logic —
this is pure build/config plumbing plus two small new files: an inline-SVG
icon-generator script (to keep us off third-party assets) and a tiny
`src/sw-register.ts` boundary that registers the SW only in the browser (no-op
under tests). `src/main.ts` gains a single import line for the registration —
all other story-006 work lives in `vite.config.ts` and `public/`.

## User Story

As an iPhone user,
I want to install the weather app to my home screen and have its shell load
offline,
So that I can check weather in one tap, including in airplane mode.

Notes:
- **Icons are committed**, not generated at build time. The generator script is
  for repeatable regeneration (e.g. when the glyph changes); it's deterministic
  and doesn't add a build-time dep. Re-run with `npm run gen:icons`.
- The manifest's icon list deliberately uses `purpose: 'any maskable'` on the
  512×512 entry. iOS ignores `maskable` (uses `apple-touch-icon`), so the
  separate 180×180 PNG is what Safari shows on the home screen — Android/Chrome
  installs use the maskable 512.
- `display: 'standalone'` is what hides the browser chrome on Add-to-Home-Screen.
- We avoid `vite-plugin-pwa`'s `injectManifest` strategy — `generateSW`
  (default) precaches everything Vite emits, which is exactly what we need for
  a static SPA shell. No custom SW logic needed (cache-on-fetch for Open-Meteo
  data is STORY-007).
- **Hotspot isolation**: `vite.config.ts` is exclusively owned by #6 in this
  wave; `src/main.ts` gets one import + one call — easy to merge if #9 also
  touches it.

### Task 3: Create the SW registration boundary

### Task 5: Generate PWA icons (committed assets)

- **File**: `scripts/generate-icons.mjs`, `public/pwa-192.png`,
  `public/pwa-512.png`, `public/apple-touch-icon-180.png`, `public/favicon.svg`
- **Action**: CREATE
- **Implement**:
  - `scripts/generate-icons.mjs` — Node ESM script (no extra deps). Writes:
    - `public/favicon.svg`: a simple sun-behind-cloud SVG using project colors
      (`--accent-warm #f5c453` sun, `--accent-cool #d9e2f0` cloud, on
      `--bg #0b1726`). ~600 bytes.
    - PNGs: rendered using Node's built-in `zlib` to deflate raw RGBA pixel
      data + a hand-rolled PNG chunk writer (IHDR, IDAT, IEND with CRC32).
      Each PNG is generated by:
      1. Allocate `width*height*4` bytes (RGBA).
      2. Fill with brand background.
      3. Stamp a circle (sun) and a rounded blob (cloud) by iterating pixels.
      4. Encode to PNG via `zlib.deflateSync` + CRC table.
    No npm deps, no `sharp`, no `canvas`. The output is committed so the build
    pipeline never has to run this script — it's a one-shot regeneration tool.
  - `package.json` scripts: `"gen:icons": "node scripts/generate-icons.mjs"`.
  - Run the script once during this task and commit the four files.
- **Mirror**: no existing precedent in the repo — this is the first
  build-tooling script. Pattern is "Node ESM script under `scripts/`", a
  conventional Node layout.
- **Validate**: `node scripts/generate-icons.mjs` exits 0; the four files
  exist; `file public/pwa-192.png` reports PNG image data 192x192; `npm run
  lint` passes (script not under `src/`, so ESLint's `*.cjs` ignore + the
  default include don't trip it — confirm by running lint).

## Risks

| Risk | Mitigation |
|------|------------|
| `vite-plugin-pwa@1.3.0` breaks against Vite 7.x. | Plugin's peerDeps explicitly include `vite ^7.0.0`. If install fails, fall back to `^0.20` (last known Vite-5+ release) — but the `1.x` line is the actively maintained one (last publish 2026-05-05). Verify via `npm install` + `npm run build` in Task 1 and 7. |
| Generating PNGs without `sharp`/`canvas` produces ugly icons. | Acceptable: this app is for one user and the PRD says "simple and recognizable (sun/cloud), no third-party assets with licensing limits". A hand-rolled raster of a solid background + a sun circle + cloud blob meets that bar. Owner can replace later by re-running `gen:icons` with a different glyph spec or by dropping in their own art. |
| `virtual:pwa-register` types not resolved → tsc errors. | Add `/// <reference types="vite-plugin-pwa/client" />` to `src/vite-env.d.ts` (Task 2). The plugin's published `client.d.ts` declares this virtual module. |
| `src/sw-register.ts` imports `virtual:pwa-register` directly → vitest can't resolve it under jsdom. | Inject the `register` function as a parameter (default to a dynamic import gated by `'serviceWorker' in navigator`). Tests pass an injected stub; production code uses the default. Static `import` statements for the virtual module are deferred via dynamic `import('virtual:pwa-register')` inside the function. |
| Hotspot conflict with #9 in `src/main.ts`. | Keep #6's main.ts diff to TWO lines (one import + one call); #9 is doing slot management. The two diffs are textually disjoint, merge should be mechanical. |
| Icons committed as binary — large repo growth. | Three PNGs at 192, 180, 512 px filled with mostly solid color → each ~1-5 KB. Total well under 50 KB. Acceptable. |
| Workbox precaches `*.png` including a future user-uploaded image. | Out of scope — no user uploads in MVP. `globPatterns` only matches files Vite emits into `dist/`. |
| The `apple-mobile-web-app-capable` meta is deprecated in favor of `mobile-web-app-capable`. | Add **both**: the new `mobile-web-app-capable` for forward compatibility, and the legacy `apple-` prefixed one for iOS Safari which still requires it. (See Apple's PWA docs — both are recommended in 2026.) |
